# 幂等性控制设计文档

## 1. 概述

本文档描述了智领软件公司基础脚手架项目的幂等性控制设计，包括幂等性控制原理、实现方式、使用方法和最佳实践。

## 2. 设计目标

- **防止重复提交**：防止用户因网络延迟、页面刷新等原因导致的重复提交
- **保证数据一致性**：确保关键业务操作只执行一次
- **用户体验优化**：提供友好的重复提交提示
- **性能优化**：使用Redis缓存提高校验效率
- **灵活配置**：支持不同接口配置不同的防重策略

## 3. 设计原理

### 3.1 幂等性控制核心思想

幂等性控制的核心思想是通过唯一标识来识别重复请求：

```
用户请求 → 生成唯一标识 → Redis检查 → 首次请求/重复请求
    ↓              ↓              ↓              ↓
  业务处理    →   存储标识    →   允许执行    →   拒绝执行
```

### 3.2 唯一标识生成策略

唯一标识由以下几部分组成：
- **用户标识**：从ThreadLocal中获取用户ID，确保不同用户隔离
- **请求参数**：对请求参数进行序列化和哈希，确保相同请求参数生成相同标识
- **接口地址**：使用请求URI作为标识的一部分，确保不同接口隔离

### 3.3 Redis存储策略

- **Key格式**：`global:repeat_submit:{接口地址}:{哈希值}`
- **Value**：空字符串或简单标识
- **过期时间**：根据注解配置的间隔时间自动过期
- **存储位置**：Redis，支持分布式环境

## 4. 技术实现

### 4.1 核心组件

#### 4.1.1 RepeatSubmit注解

```java
@Inherited
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface RepeatSubmit {
    // 防重间隔时间（默认5000ms）
    int interval() default 5000;
    // 时间单位（默认毫秒）
    TimeUnit timeUnit() default TimeUnit.MILLISECONDS;
    // 重复提交提示语（支持国际化）
    String message() default "{repeat.submit.message}";
}
```

#### 4.1.2 RepeatSubmitAspect切面类

切面类实现了幂等性控制的核心逻辑：
- **前置通知**：在方法执行前进行幂等性校验
- **后置通知**：在方法执行后根据结果决定是否保留标识
- **异常通知**：在方法抛出异常时删除标识，允许重试

#### 4.1.3 IdempotentConfig配置类

自动配置切面类，确保Spring容器中存在RepeatSubmitAspect的Bean。

### 4.2 实现流程

#### 4.2.1 前置校验流程

```
1. 获取注解配置的间隔时间
2. 校验间隔时间不小于1秒
3. 获取当前请求的HttpServletRequest
4. 序列化请求参数（过滤无效参数）
5. 获取请求URI
6. 从ThreadLocal获取用户ID
7. 生成MD5哈希值
8. 构建Redis Key
9. 检查Redis中是否存在Key
   - 存在：抛出重复提交异常
   - 不存在：存储Key并设置过期时间
```

#### 4.2.2 后置处理流程

```
1. 获取方法返回结果
2. 判断结果是否为Result类型
3. 检查结果状态码
   - 成功（200）：保留Redis Key，防止重复提交
   - 失败：删除Redis Key，允许重试
4. 清理ThreadLocal中的Key
```

#### 4.2.3 异常处理流程

```
1. 捕获方法抛出的异常
2. 删除Redis中的Key
3. 清理ThreadLocal中的Key
4. 允许异常重试
```

### 4.3 关键技术点

#### 4.3.1 参数序列化与过滤

```java
private String argsArrayToString(Object[] paramsArray) {
    StringBuilder params = new StringBuilder();
    if (paramsArray != null && paramsArray.length > 0) {
        for (Object o : paramsArray) {
            if (ObjectUtil.isNotNull(o) && !isFilterObject(o)) {
                try {
                    params.append(JsonUtils.toJsonString(o)).append(" ");
                } catch (Exception e) {
                    log.warn("参数序列化失败", e);
                }
            }
        }
    }
    return params.toString().trim();
}
```

#### 4.3.2 无效参数过滤

过滤以下类型的参数，避免序列化问题：
- MultipartFile及其数组/集合
- HttpServletRequest
- HttpServletResponse
- BindingResult

#### 4.3.3 用户标识获取

从ThreadLocal中获取用户ID，确保在认证拦截器执行后获取到用户信息：

```java
Long userId = UserThreadLocal.getUserId();
String submitKey = userId != null ? String.valueOf(userId) : "anonymous";
```

#### 4.3.4 Redis操作

直接使用Spring的RedisTemplate进行Redis操作，避免引入额外的工具类：

```java
private RedisTemplate<String, Object> getRedisTemplate() {
    return SpringUtil.getBean("redisTemplate");
}
```

## 5. 使用方式

### 5.1 基本使用

在需要防重提交的Controller方法上添加@RepeatSubmit注解：

```java
@PostMapping("/save")
@RepeatSubmit(interval = 5000, timeUnit = TimeUnit.MILLISECONDS)
public Result<Void> saveUser(@RequestBody UserDTO userDTO) {
    userService.save(userDTO);
    return Result.success();
}
```

### 5.2 自定义配置

```java
@PostMapping("/submit")
@RepeatSubmit(
    interval = 10, 
    timeUnit = TimeUnit.SECONDS,
    message = "{custom.submit.repeat.message}"
)
public Result<Void> submitForm(@RequestBody FormDTO formDTO) {
    formService.submit(formDTO);
    return Result.success();
}
```

### 5.3 国际化消息

在国际化配置文件中添加消息：

```properties
# messages.properties
repeat.submit.message=请勿重复提交，{0}秒后再试
custom.submit.repeat.message=表单已提交，请勿重复操作

# messages_en.properties
repeat.submit.message=Please do not submit repeatedly, try again after {0} seconds
custom.submit.repeat.message=Form has been submitted, please do not repeat
```

## 6. 最佳实践

### 6.1 适用场景

幂等性控制适用于以下场景：
- **表单提交**：防止用户重复点击提交按钮
- **支付操作**：防止重复扣款
- **订单创建**：防止创建重复订单
- **数据修改**：防止重复修改导致数据不一致
- **文件上传**：防止重复上传相同文件

### 6.2 不适用场景

以下场景不建议使用幂等性控制：
- **查询操作**：查询操作本身是幂等的
- **数据导出**：导出操作通常允许重复执行
- **实时数据刷新**：需要实时获取最新数据的操作

### 6.3 配置建议

1. **间隔时间设置**：
   - 普通表单提交：3-5秒
   - 支付操作：10-30秒
   - 重要业务操作：根据业务需求设置

2. **提示消息**：
   - 提供明确的错误提示
   - 包含重试时间信息
   - 支持国际化

3. **异常处理**：
   - 记录重复提交日志
   - 提供友好的用户提示
   - 避免暴露系统内部信息

### 6.4 性能优化

1. **Redis优化**：
   - 使用连接池
   - 设置合理的过期时间
   - 避免存储大对象

2. **参数序列化优化**：
   - 过滤不必要的参数
   - 使用高效的JSON库
   - 避免循环引用

3. **日志优化**：
   - 只在必要时记录详细日志
   - 避免在高频接口上记录过多日志
   - 使用异步日志

## 7. 扩展指南

### 7.1 自定义幂等性策略

如果需要实现更复杂的幂等性控制，可以：

1. **扩展RepeatSubmit注解**：
   ```java
   public @interface RepeatSubmit {
       // 现有属性...
       
       // 新增属性
       String keyGenerator() default "";
       boolean includeHeaders() default false;
       String[] excludeParams() default {};
   }
   ```

2. **实现自定义Key生成器**：
   ```java
   public interface IdempotentKeyGenerator {
       String generate(JoinPoint point, RepeatSubmit annotation);
   }
   ```

3. **扩展切面逻辑**：
   ```java
   @Before("@annotation(repeatSubmit)")
   public void doBefore(JoinPoint point, RepeatSubmit repeatSubmit) throws Throwable {
       // 自定义幂等性校验逻辑
   }
   ```

### 7.2 集成其他存储

除了Redis，还可以集成其他存储：

1. **数据库存储**：
   - 创建幂等性记录表
   - 使用数据库事务保证一致性
   - 适合单机应用

2. **本地缓存**：
   - 使用Caffeine或Guava Cache
   - 适合单机应用
   - 性能更高但不支持分布式

3. **分布式锁**：
   - 使用Redisson或Redis分布式锁
   - 适合高并发场景
   - 可以实现更复杂的控制逻辑

## 8. 监控与告警

### 8.1 监控指标

1. **重复提交率**：
   - 监控重复请求占总请求的比例
   - 设置阈值告警

2. **响应时间**：
   - 监控幂等性校验的耗时
   - 确保不影响接口性能

3. **Redis使用情况**：
   - 监控Key的数量和过期时间
   - 避免内存泄漏

### 8.2 告警策略

1. **重复提交率过高**：
   - 可能表明用户界面问题
   - 可能表明网络问题

2. **幂等性校验失败**：
   - Redis连接问题
   - 序列化问题

3. **性能问题**：
   - 校验耗时过长
   - 影响用户体验

## 9. 故障排查

### 9.1 常见问题

1. **幂等性校验不生效**：
   - 检查注解是否正确添加
   - 检查切面是否被Spring代理
   - 检查AOP配置是否正确

2. **用户信息获取失败**：
   - 检查ThreadLocal中是否有用户信息
   - 检查认证拦截器执行顺序
   - 检查用户登录状态

3. **Redis连接问题**：
   - 检查Redis配置
   - 检查网络连接
   - 检查Redis服务状态

### 9.2 调试技巧

1. **开启详细日志**：
   ```properties
   logging.level.com.zl.idempotent.aspectj.RepeatSubmitAspect=DEBUG
   ```

2. **检查Redis数据**：
   ```bash
   redis-cli keys "global:repeat_submit:*"
   redis-cli get "global:repeat_submit:/api/user/save:abc123"
   ```

3. **分析请求参数**：
   - 检查参数序列化结果
   - 检查哈希值生成
   - 检查Key构建逻辑

## 10. 总结

幂等性控制是保证系统数据一致性和用户体验的重要机制。本设计通过AOP切面、Redis缓存和ThreadLocal用户隔离，实现了灵活、高效的幂等性控制方案。

关键优势：
- **非侵入式**：通过注解实现，不影响业务代码
- **分布式支持**：使用Redis存储，支持集群部署
- **用户隔离**：基于用户ID进行隔离，确保多用户环境正确性
- **灵活配置**：支持不同接口配置不同的防重策略
- **性能优化**：使用高效的序列化和缓存策略

通过合理使用幂等性控制，可以有效防止重复提交问题，提高系统的健壮性和用户体验。
