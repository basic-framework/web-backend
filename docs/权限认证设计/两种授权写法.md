# Spring Security 配置说明：@EnableWebSecurity 与两种授权写法对比

## 一、@EnableWebSecurity 注解作用

`@EnableWebSecurity` 是 Spring Security 提供的核心注解，用于**显式启用 Web 安全功能**，其核心作用包括：

1. **自动装配安全组件**：触发 Spring Security 核心 Bean 的自动配置（如 `SecurityFilterChain`、`AuthenticationManager`、`CsrfFilter` 等），无需手动注册这些组件。
2. **启用安全过滤链**：将 Spring Security 的过滤链（`SecurityFilterChain`）集成到 Servlet 容器的过滤链中，实现对 HTTP 请求的安全拦截。
3. **支持自定义配置**：配合 `@Configuration` 注解，允许开发者通过 `HttpSecurity` 等对象自定义安全规则（如权限控制、认证方式、跨域等）。
4. **版本兼容性**：在 Spring Security 6.x 及以上版本中，推荐显式添加该注解（旧版隐式启用，新版更强调显式配置，增强可读性）。

## 二、两种授权写法对比

Spring Security 提供了「过滤链级全局授权」和「方法级细粒度授权」两种核心授权方式，以下是详细对比：

### 1. 写法一：过滤链级全局授权（`.anyRequest().access(authorizationManager)`）

#### 核心代码示例

java



运行









```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(authorize -> authorize
            .requestMatchers("/login").permitAll()
            // 全局授权：所有请求由 authorizationManager 统一校验
            .anyRequest().access(jwtAuthorizationManager)
        );
    return http.build();
}

// 定义全局授权管理器
@Bean
public AuthorizationManager<RequestAuthorizationContext> jwtAuthorizationManager() {
    // 配置路径与权限的映射规则
    RequestMatcherDelegatingAuthorizationManager manager = new RequestMatcherDelegatingAuthorizationManager();
    manager.add(RequestMatchers.antMatchers("/admin/**"), AuthorizationManagers.hasRole("ADMIN"));
    manager.add(RequestMatchers.antMatchers("/user/**"), AuthorizationManagers.hasAnyRole("ADMIN", "USER"));
    manager.setDefaultAuthorizationManager(AuthorizationManagers.authenticated());
    return manager;
}
```

### 2. 写法二：方法级细粒度授权（`.anyRequest().authenticated()` + `@PreAuthorize`）

#### 核心代码示例

java



运行









```java
// 1. 启用方法级授权
@EnableGlobalMethodSecurity(prePostEnabled = true)
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(authorize -> authorize
                .requestMatchers("/login").permitAll()
                // 过滤链仅校验“是否认证”，授权交给方法注解
                .anyRequest().authenticated()
            );
        return http.build();
    }
}

// 2. Controller 中配置方法级授权
@RestController
@RequestMapping("/admin")
public class AdminController {
    // 仅 ADMIN 角色可访问
    @GetMapping("/list")
    @PreAuthorize("hasRole('ADMIN')")
    public Result<List<User>> getUserList() {
        // 业务逻辑
    }
}
```

### 3. 两种写法对比表

| 对比维度       | 过滤链级全局授权                         | 方法级细粒度授权                            |
| -------------- | ---------------------------------------- | ------------------------------------------- |
| **授权时机**   | 请求到达 Controller 前，过滤链拦截时授权 | 请求进入 Controller 方法时授权              |
| **授权粒度**   | 全局统一规则（按路径批量配置）           | 接口 / 方法级细粒度（单个接口独立配置）     |
| **灵活性**     | 低：需在授权管理器中维护路径 - 权限映射  | 高：每个接口可单独配置差异化权限            |
| **职责边界**   | 认证、授权耦合在过滤链中                 | 过滤链负责 “认证”，方法负责 “授权”，解耦    |
| **可维护性**   | 差：权限规则集中管理，接口增多后易臃肿   | 好：权限与接口绑定，修改时仅需调整注解      |
| **适用场景**   | 权限规则简单统一（如仅区分 ADMIN/USER）  | 权限差异化明显的复杂系统（前后端分离项目）  |
| **代码侵入性** | 低：无需修改业务代码，仅需维护授权管理器 | 中：需在业务方法上添加 `@PreAuthorize` 注解 |

## 三、总结

1. `@EnableWebSecurity` 是启用 Spring Security Web 安全功能的核心注解，负责装配安全组件与过滤链。
2. **过滤链级授权**适合简单场景，通过全局管理器统一管控权限；**方法级授权**适合复杂场景，通过注解实现接口差异化权限。
3. 实际项目中，推荐优先使用「方法级细粒度授权」，结合过滤链的「认证校验」，实现 “认证在过滤链、授权在方法” 的清晰职责划分。
